### Задание 1. Резервное копирование

### Кейс
Финансовая компания решила увеличить надёжность работы баз данных и их резервного копирования. 

Необходимо описать, какие варианты резервного копирования подходят в случаях: 

1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.

1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.

1.3.* Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных.

*Приведите ответ в свободной форме.*

---
### ОТВЕТ на Задание 1.

Финансовой компании в основном стремятся повысить надёжность работы баз данных и их резервное копирование для сохранения прибыли. Для них важно выбрать методы в зависимости от конкретных требований к восстановлению данных.

1.1. Необходимость восстановливать данные в полном объёме за предыдущий день.
Для этого случая подходит полное резервное копирование (Full Backup). При полном бэкапе создаётся точная копия всей базы данных, включая все таблицы, индексы, настройки и другие элементы. Это позволяет восстановить данные до состояния на момент создания резервной копии. 

Преимуществом можно считать простоту восстановления, где требуется только одна резервная версия, а так же гарантированное восстановление всей базы данных.
А вот неодостатки такоего метода - это как затратность по времени и ресурсам, особенно для больших объёмов данных, так и потребление значительного объёма дискового пространства.
В этом случае необходимо выполнять полное копирование регулярно (ежедневно), а для динамичных систем можно комбинировать с дифференциальным или инкрементным копированием для снижения нагрузки.

1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.

Преполага, что в этом случае подойдут два варианта:

- 1. Инкрементное резервное копирование. После ежедневного полного бэкапа каждый час копируются только те данные, которые изменились с момента последнего бэкапа (независимо от его типа). Это снизит нагрузку на жёсткие диски и ускорит процесс копирования, но для восстановления потребуется последовательное применение всех инкрементных копий вместе с полной резервной версией. 
- 2. Непрерывное резервное копирование. Каждая транзакция или изменение в базе данных немедленно реплицируется на резервный сервер. Это позволяет восстановить данные почти в реальном времени, включая состояние за час до сбоя.

1.3. Моментальное переключение на работающую или починенную базу данных.

Да, такой сценарий возможен благодаря технологиям высокой доступности. Для реализации моментального переключения используются:

- Репликация в реальном времени. Синхронная или асинхронная репликация изменений на вторичный сервер обеспечивает наличие актуальной копии данных, готовой к переключению в случае сбоя основного сервера. При синхронной репликации данные на всех копиях всегда согласованы, но это может увеличивать задержки записи. Асинхронная репликация допускает временную задержку и возможные рассогласования. 
- Кластеризация баз данных. Группа серверов работает так, что при выходе из строя одного из них другой автоматически берёт на себя его функции. 
 
По итогу, выбор решения зависит от требований бизнеса к доступности данных, допустимому объёму потери данных, бюджета и технической инфраструктуры.

### ОТВЕТ на Задание 1.
2.1 Пример команд резервирования и восстановления БД (pgdump/pgrestore).

- Пример команды для создания резервной копии базы данных с помощью [оффициальной документации - pg_dump](https://www.postgresql.org/docs/current/app-pgdump.html):

```
pg_dump -U username -d database_name -F c -f backup.dump
```
Где:
`pg_dump - утилита для экспорта базы данных`
`-U username - имя для подключения к PostgreSQL;`
`-d database_name - имя базы, которой нужно резервное скопировать;`
`-F c - выбор формата вывода (custom, архивный формат);`
`-f backup.dump - имя файла, в который сохранится дамп.`

- Пример команды для восстановления базы данных с помощью [оффициальной документации - pg_restore](https://www.postgresql.org/docs/18/app-pgrestore.html):

```
pg_restore -U username -d database_name -c backup.dump
```
Где:
`pg_restore - утилита для импорта баз данных`
`-U username - имя для подключения к PostgreSQL;`
`-d database_name -  имя базы, которой восстанавливаю данные;`
`-c - опция, которая удаляет существующие объекты базы данных перед восстановлением (обеспечивает «чистое» восстановление);`
`backup.dump - имя файла с резервной копией.`

2.2 Автоматизация этого процесса.

Процесс резервного копирования и восстановления PostgreSQL можно автоматизировать используя планировщики задач (например, cron).


bash-скрипт `pgsql_backup.sh` для создания дампа:

```
#!/bin/bash
TIMESTAMP=$(date +%F-%H-%M)
BACKUP_DIR="/var/backups/pgsql"
DB_NAME="mydb"
USER="postgres"
mkdir -p $BACKUP_DIR
pg_dump -U $USER $DB_NAME > $BACKUP_DIR/${DB_NAME}_${TIMESTAMP}.sql
```

Добавляем задачу в crontab, чтобы скрипт запускался по расписанию (например, каждый день в 02:00):
```
crontab -e
0 2 * * * /path/to/pgsql_backup.sh 
```

### ОТВЕТ на Задание 3.

3.1. С помощью официальной документации приведите пример команды инкрементного резервного копирования базы данных MySQL.

Инкрементное резервное копирование в MySQL можно выполнять с помощью утилиты mysqlbackup (входит в MySQL Enterprise Edition) или на основе бинарных журналов.

В [оффициальной документации](https://dev.mysql.com/doc/mysql-enterprise-backup/4.1/en/) MySQL Enterprise Backup приведены два варианта инкрементного резервного копирования с помощью mysqlbackup:

- С использованием опции `--incremental-base`. 
Эта опция позволяет не отслеживать значения LSN между резервными копиями. 
Можно запросить у `mysqlbackup` значение `end_lsn` из последней успешной резервной копии (не TTS), записанной в таблице `backup_history` на сервере, используя `--incremental-base=history:last_backup` или `history:last_full_backup` (для версий 8.0.17 и позже).

Пример команды:
```
mysqlbackup --defaults-file=/home/dbadmin/my.cnf \
  --incremental \
  --incremental-base=history:last_backup \
  --backup-dir=/home/dbadmin/temp_dir \
  --backup-image=incremental_image1.bi \
  backup-to-image
```
- С использованием опции `--start-lsn`. 
Позволяет указать, с какого значения должно начинаться инкрементное резервное копирование. Нужно записать LSN предыдущей резервной копии, который `mysqlbackup` сообщает в конце резервного копирования. Затем этот номер указывается в `mysqlbackup` с помощью `--start-lsn`. Инкрементная резервная копия включает все изменения, произошедшие после указанного LSN.

Пример команды:
```
mysqlbackup --defaults-file=/home/dbadmin/my.cnf --incremental \
  --start-lsn=2654255716 \
  --with-timestamp \
  --backup-dir=/incr-tmp \
  --backup-image=/incr-backup/incremental_image.bi \
  backup-to-image
```
Инкрементное резервное копирование на основе бинарных журналов.
Этот метод предполагает, что первая резервная копия в цепочке должна быть полной. Для создания инкрементной копии копируются только новые файлы бинарных журналов, которые не были включены в предыдущую резервную копию. 

Тут важно понимать, что перед созданием инкрементной копии необходимо включить ведение бинарных журналов. 

Пример команды для создания полной резервной копии с использованием mysqldump:
```
mysqldump -uroot -p --all-databases --single-transaction --flush-logs --master-data=2 > full_backup.sql
```
Здесь `--flush-logs` инициализирует запись нового файла бинарного журнала, а `--master-data=2` записывает имя файла бинарного журнала в дамп. 
Для инкрементного копирования после создания полной резервной копии и модификации данных можно выполнить команды в MySQL Command Line Client:
```
flush binary logs;
show binary logs;
```
Затем скопировать файлы бинарных журналов в место хранения резервных копий. После копирования файлы следует удалить, чтобы при следующем копировании они не дублировали уже созданные резервные копии данных. 

3.2. В каких случаях использование реплики будет давать преимущество по сравению с обычным резервным копированием?

Репликация базы данных предпочтительнее обычного резервного копирования в сценариях, где критичны высокая доступность (реплика может заменить вышедший из строя основной сервер), масштабирование нагрузки (распределение операций чтения на реплики), оперативное восстановление данных (копии всегда актуальны), снижение сетевых задержек (размещение реплик ближе к пользователям), параллельная работа с данными (выполнение отчётов и аналитики без влияния на основной сервер) и защита от человеческих ошибок (благодаря отложенной репликации). 
При этом репикация требует более сложной настройки, может порождать проблемы с консистентностью данных и риск потери последних транзакций при асинхронной репликации — поэтому для долгосрочного архивирования и соблюдения нормативных требований лучше подходит традиционное резервное копирование.
